# CLAUDE.md — Engawa development rules

Standing rules. These survive the entire project lifetime; they are not a plan. (`AGENTS.md` mirrors this file for Codex-family agents — keep the two byte-identical.)

## Repo layout

```
engawa/
  spec/              # THE CONTRACT — normative; DRAFT until tagged contract-1.0.0
    contract.md
    commands/        # per-namespace normative detail, one file per namespace
    errors.md        # error-code registry
    assets.md        # app:// resolution rules
  shell-js/          # shared JS runtime; identical bytes on all hosts
  conformance/       # suite + runner + Node mock host
  hosts/
    macos/           # Swift reference host
    windows/         # empty until contract freeze; clean-room rule applies
    linux/           # empty until contract freeze; clean-room rule applies
  adapters/
    sqlite/          # reference adapter; extractable — mirrors external adapter layout exactly
    update/          # contract-coupled adapter; implements §7.1 + §8, versions with the contract, never extracted
  examples/
    notes/           # acceptance-gate app
  docs/design.md     # decision record; read for rationale, do not "update" it
  .claude/commands/  # slash commands provided to adapter authors
```

## Definition of done — any change

1. `make conformance` green (suite against macOS host AND mock host), and
2. `make notes` passes its full scripted sequence: build host with `sqlite` + `update` adapters → bundle `examples/notes` → launch → write one record → quit → relaunch → read it back → serve a locally signed app-update manifest → update applies → relaunch shows the updated asset with the record intact.

Conformance-green alone is NOT completion. Never report done without the gate.

Gate mechanics (fixed decisions — do not re-open):
- `examples/notes` runs in autotest mode when launched with `ENGAWA_AUTOTEST=1`: it performs the write/read/update steps itself and reports via exit code + a result file. No UI scripting, no AppleScript.
- Dev signing keypair (ed25519) is generated by the Makefile on first build and embedded as the trust root for dev builds; the suite and the gate sign with it. Release keys are out of scope until distribution.
- macOS code signing in dev and CI is ad-hoc (`codesign -s -`). Notarization is a stub hook.

## Commit discipline

- A command or adapter change lands **spec section + conformance test + implementation in one commit**. Divergence among the three is this architecture's failure mode; the discipline makes it impossible rather than discouraged.
- Ambiguity found during implementation: the resolution goes into `spec/`. Never into host-specific behavior, never into a code comment, never into this file.

## Spec status

`spec/` is DRAFT and grows with implementation until tagged `contract-1.0.0`. After the tag: additive-only within the major version; the handshake's `contractVersion` carries negotiation.

## Cross-host discipline (supersedes the clean-room rule)

The clean-room rule — Windows/Linux implemented without reading `hosts/macos/` source — did its job: it bootstrapped three independent hosts and shook the spec holes out during initial implementation. It is **retired**. Going forward, cross-host consistency is held by two standing disciplines, not by author blindness:
- **Spec-first**: a behavior lands in `spec/` (and a conformance test) before any host implements it. Ambiguity resolves in `spec/`, never in host-specific behavior.
- **Conformance-green on every host**: the one suite runs against all three real hosts (and the mock); a cross-host change is not done until it is green on every leg. CI (`.github/workflows`) enforces this on every change.

Note: `update` is not an ordinary adapter. Contract §7.1 (signature verification before anything lands under `app://`, embedded trust root) and the atomic asset-root swap are **host obligations** — a host is non-conformant without them, regardless of adapter presence.

## Non-negotiables (from the contract — repeated here because agents skim)

- Binary never rides the message channel; use the `app://io` token flow.
- Events never carry bulk payloads; streams are pull.
- One eval per main-loop tick for delivery; batch frames.
- `window.closeRequested` has no timeout.
- No dynamic adapter loading. No localhost server. No privileged dispatch path for built-ins.
- `process.spawn` only runs manifest-declared, in-bundle sidecars (§7.2). Default CSP per §7.3; relaxations only via `engawa.json`.

## Autonomous execution

Decide work order, test methodology, and infrastructure design yourself. Record the choice and its rationale in the commit message, and proceed — no prior confirmation.

Stop and ask only when:
- (a) a normative part of the contract (a MUST / MUST NOT in `spec/`) must be changed or relaxed;
- (b) a fixed decision (the gate mechanics in this file) would be contradicted;
- (c) the gate definitions themselves (`make notes` / `make conformance`) need to change.

Otherwise, when in doubt: decide, record it in `spec/` or the commit message, and proceed.
